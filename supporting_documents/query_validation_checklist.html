1
STEP 1 — WIKI INDEX SEARCH
store.readIndex() → S3 GetObject
Testbed: fs.readFileSync("wiki/index.md")
Cloud: s3.GetObject({Key: "wiki/index.md"})
Returns the same string. searchWikiIndex() receives identical input. No change needed.
2
STEP 2 — READ WIKI PAGES + EXTRACT POINTERS
store.readWikiPage(relPath) → S3 GetObject
Testbed: fs.readFileSync("wiki/entities/gandalf.md")
Cloud: s3.GetObject({Key: "wiki/entities/gandalf.md"})
Returns the same markdown string. parseWikiFrontmatter() is pure string parsing — no I/O dependency. Source pointers extracted identically.
3
STEP 3 — TARGETED KB RETRIEVAL
kb.retrieveTargeted() — s3_key path format must match
Testbed: wiki pointers use tolkien/raw/silmarillion.pdf
Cloud: wiki pointers must use compartments/corp-docs/raw/silmarillion.pdf
The s3_key in wiki frontmatter must match the actual S3 key that Bedrock KB indexed. If your AD sync lands files at a different prefix than what the testbed used, the ingest pipeline needs to write the correct s3_key into frontmatter. This is the one thing to verify.
4
STEP 4 — FORMAT DUAL CONTEXT BLOCKS
budgetContext() — pure computation, no I/O
Takes wiki page strings + RAG chunk objects, returns trimmed context blocks. 60/40 budget split. No dependency on where data came from. Identical behavior.
5
STEP 5 — BEDROCK CALL WITH DUAL INPUT
client.hybridQuery() → InvokeModelCommand
Testbed: fetch("https://api.anthropic.com/v1/messages", {system, messages})
Cloud: bedrock.send(new InvokeModelCommand({body: {system, messages}}))
Same prompt structure. Same system prompt. Same message format. Both wiki context AND RAG context are in the user message — this is your dual-input pattern. Bedrock's Anthropic-compatible endpoint accepts the identical payload.
6
STEP 6 — STRUCTURED RESULT
Return { answer, method, wikiPagesUsed, sourcesConsulted }
Response parsing is identical — both Anthropic API and Bedrock return content[0].text. The structured result object is assembled from local variables. No change needed.

STALENESS DETECTION
detectStaleness() — pure comparison, no I/O
Compares wiki pointer hashes against RAG chunk source hashes. Both are already in memory by step 3. No dependency on storage backend.
+
MULTI-TURN CONVERSATION
ConversationMessage[] — caller-managed, passes through
History array is passed in by the caller and appended to the Bedrock messages array. No storage dependency. Works identically whether called from CLI or Lambda.
